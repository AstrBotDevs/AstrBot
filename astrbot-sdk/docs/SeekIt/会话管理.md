# 会话管理

## 两个核心概念，一个装饰器：

ConversationSession - 用于构建多轮对话流。通过 await session.ask("问题") 等待用户回复，超时或被新会话替换时自动关闭。典型场景：表单填写、游戏交互。

SessionWaiterManager - 底层会话等待器，通过 ctx.register_task() 后台运行，避免阻塞当前消息处理。SDK 会自动警告不当使用。

@session_waiter 装饰器 - 将普通 handler 转换为会话式 handler，自动管理超时和状态。通过 SessionController 控制会话生命周期（controller.keep() / controller.stop()）。

## 关键规则：

永远使用 await ctx.register_task(waiter(...), "task-name") 启动会话等待器，不要直接 await，否则会阻塞消息分发管线
会话按 unified_msg_origin 隔离，同一会话只能有一个活跃的等待器
超时默认 30 秒，可配置；超时或取消会抛出异常


## 高级开发需要了解的（运行时层面）
你需要理解整个会话生命周期管线：

### 会话键与隔离

_WaiterKey = tuple[plugin_id, session_key]，每个插件可以在同一会话上注册独立的等待器
支持 ConversationState 状态机：ACTIVE → TIMEOUT | REPLACED | COMPLETED | CANCELLED
ConversationReplaced 异常用于处理新会话抢占旧会话的场景

### 任务跟踪与警告机制

SDK 使用 WeakSet 跟踪 handler 任务和后台任务（
session_waiter.py#L42-L44L42-L44）
直接在 handler 中 await 会触发警告："Direct await on session_waiter blocks the current handler dispatch"
通过 _TaskReentrantLock 防止同一会话并发操作导致状态损坏

### 消息历史链追踪

record_history_chains=True 时，SessionController.history_chains 记录所有中间消息，用于 LLM 上下文重建
参见 
SessionController.get_history_chains()L175-L176
### 能力分发集成

等待器通过 _invoke_system_waiter 调用 system.session_waiter_* 能力（注册/注销/分发）
运行时通过 dispatcher.dispatch(event) 桥接到具体等待器，参见 
SessionWaiterManager.dispatch()L395-L448
### 取消与超时传播

ConversationSession 绑定 _owner_task，跨任务使用会抛出 ConversationClosed
CancelToken 通过 Context 传递，用于处理请求级别的取消信号（
context.py#L194-L226L194-L226）
### 反模式警示：

不要在普通 handler 内直接 await 会话等待器（会挂起整个 dispatch）
不要跨任务共享 ConversationSession 实例（runtime 会检测并抛异常）
不要在会话关闭后继续调用 ask() 或 reply()（状态检查会拒绝
