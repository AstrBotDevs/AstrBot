# 人格(Persona)继承修复思路

## 问题描述

### 现象
用户在对话中设置了人格（persona）后，执行以下任一操作创建新对话时，新对话的 `persona_id` 为 `NULL`：
- 执行 `/new` 命令创建新对话
- 在没有当前对话的情况下发送第一条消息
- 其他触发 `new_conversation()` 的场景

### 影响
人格设置丢失，用户需要重新设置人格，影响用户体验。

## 根本原因分析

### 当前代码逻辑
```python
# conversation_mgr.py
async def new_conversation(
    self,
    unified_msg_origin: str,
    platform_id: str | None = None,
    content: list[dict] | None = None,
    title: str | None = None,
    persona_id: str | None = None,  # ← 参数存在，但大多数调用点未传递
) -> str:
    if not platform_id:
        parts = unified_msg_origin.split(":")
        if len(parts) >= 3:
            platform_id = parts[0]
    if not platform_id:
        platform_id = "unknown"

    conv = await self.db.create_conversation(
        user_id=unified_msg_origin,
        platform_id=platform_id,
        content=content,
        title=title,
        persona_id=persona_id,  # ← 如果未传递，则为 None
    )
    # ...
```

### 问题根源
1. `new_conversation()` 方法虽然接受 `persona_id` 参数，但**所有调用点都未传递**该参数
2. 没有任何逻辑在创建新对话时"继承"当前对话的人格设置
3. 导致新对话的 `persona_id` 始终为 `NULL`

## 解决方案

### 核心思路
在创建新对话时，自动获取并继承当前对话的人格设置。

### 实现步骤

#### 步骤1：添加辅助方法获取当前人格

在 `ConversationManager` 中添加 `get_current_persona_id()` 方法：

```python
# astrbot/core/conversation_mgr.py

async def get_current_persona_id(self, unified_msg_origin: str) -> str | None:
    """获取当前对话的persona_id，用于创建新对话时继承。

    Args:
        unified_msg_origin: 统一的消息来源字符串

    Returns:
        当前对话的persona_id，如果没有或为"[%None]"则返回None
    """
    curr_cid = await self.get_curr_conversation_id(unified_msg_origin)
    if curr_cid:
        conv = await self.db.get_conversation_by_id(cid=curr_cid)
        if conv and conv.persona_id and conv.persona_id != "[%None]":
            return conv.persona_id
    return None
```

**关键点**：
- 检查当前对话是否存在
- 如果 `persona_id` 为特殊标记 `"[%None]"`（表示用户明确清除人格），则返回 `None`
- 否则返回当前对话的 `persona_id`

## 人格优先级说明

### 创建新对话时的优先级

```
1. 自定义规则（session_service_config.persona_id）  ← 最高优先级
   ↓ (如果不存在)
2. 当前对话的人格（conversation.persona_id）      ← 继承逻辑
   ↓ (如果不存在或为 [%None])
3. NULL（不设置任何人格）
```

### 后续使用对话时的优先级

在 `PersonaManager.resolve_selected_persona()` 中处理：

```
1. 自定义规则（session_service_config.persona_id）  ← 最高优先级
   ↓ (如果不存在)
2. 对话的人格（conversation.persona_id）          ← 从数据库读取
   ↓ (如果不存在)
3. 配置默认人格（provider_settings.default_personality）
   ↓ (如果不存在)
4. 无人格
```

## 特殊标记说明

### `[%None]` 的语义

- `NULL` 或 `None`：创建时未传递，需动态解析
- `"具体ID"`：用户明确设置或自动继承的人格
- `"[%None]"`：用户明确清除人格，**不应用任何人格**

**示例**：
```python
# 用户执行 /persona none 清除人格
await update_conversation_persona_id(umo, "[%None]")

# 创建新对话时
persona_id = await get_current_persona_id(umo)  # → 返回 None
await new_conversation(umo, persona_id=persona_id)  # → 新对话也为 None
```

## 测试场景

### 场景1：正常继承
1. 用户在对话A中执行 `/persona psychologist`
2. 执行 `/new` 创建对话B
3. ✅ 对话B的 `persona_id` 应为 `"psychologist"`

### 场景2：清除人格后不继承
1. 用户在对话A中执行 `/persona none`
2. 执行 `/new` 创建对话B
3. ✅ 对话B的 `persona_id` 应为 `NULL`（不继承任何人格）

### 场景3：首次创建对话
1. 新用户首次发送消息
2. ✅ 创建的对话 `persona_id` 应为 `NULL`
3. 后续使用时会回退到配置默认人格

### 场景4：自定义规则优先
1. 用户设置了自定义规则（强制应用人格 "assistant"）
2. 当前对话的人格为 "psychologist"
3. 执行 `/new` 创建���对话
4. ✅ 新对话应继承 "psychologist"（自定义规则在解析阶段生效）

## 性能影响

### 数据库查询
- 有当前对话：1次数据库查询（`get_conversation_by_id`）
- 无当前对话：0次额外查询

### 优化建议
- `get_curr_conversation_id()` 有内存缓存，性能开销小
- 只在创建新对话时执行，不影响消息处理主流程
